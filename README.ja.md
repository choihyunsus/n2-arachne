## 課題 — なぜAIはあなたのコードを正しく理解できないのか

医者に行って**「頭が痛いです」**と言ったとします。

- **ダメな医者**: 500ページの全病歴を読み、混乱し、間違った薬を処方する
- **良い医者**: 関連する記録だけ — 最近の症状、服薬歴、アレルギー — を確認し、的確に診断する

**AIコーディングアシスタントは、あのダメな医者と同じです。**

プロジェクトに500ファイルあると、AIはそのすべてを読めません。では何が起きるか？

```
 あなたのプロジェクト (500ファイル, 200万トークン)
│
├── auth/login.ts ← バグはここにある
├── auth/session.ts ← loginがインポートしている
├── api/http.ts ← sessionがインポートしている
├── utils/config.ts ← タイムアウト設定がここにある
│
├── pages/home.tsx ← 完全に無関係
├── pages/about.tsx ← 完全に無関係
├── components/Button.tsx ← 完全に無関係
└── ... 他493ファイル ← すべて無関係
```

| アプローチ | AIが受け取るもの | 結果 |
|----------|-----------------|--------|
| 全部渡す | 2,000,000トークン | コンテキストウィンドウを超過、AI混乱 |
| ランダムなファイル | 約50,000トークン | 重要なコードを見逃し、誤った修正 |
| **Arachne** | **30,000トークン** (関連4ファイル) | 毎回正確な修正 |

> **トークン** = AIが読むテキストの単位。トークンが多い = コスト増、低速化、精度低下。
> AIには限られた「コンテキストウィンドウ」があります — 一度に広げられる書類の量に限りがある机のようなものです。

### 実環境ベンチマーク (N2 Browser — 3,219ファイル)

| 指標 | 値 |
|--------|:-----:|
| **プロジェクトサイズ** | 3,219ファイル, **468万トークン** |
| **Arachne出力** | **14,074トークン** |
| **圧縮率** | **333倍** (99.7%削減) |
| **インデックス時間** | 627ms (増分: 0ms) |
| **DBサイズ** | 24 MB |

> *実際のプロダクションプロジェクトでの計測結果。Arachneは必要なものだけを正確に提供 — データ量333分の1で同じ精度。*

## 解決策 — ArachneはAIに必要なものだけを選ぶ

Arachneは**ローカルMCPサーバー**で、あの良い医者のように機能します。コードベース全体を一度読み取り、構造を理解し、**関連するものだけ**をAIに送信します。

```
あなた: "ログインのタイムアウトバグを直して"
 │
 ▼
┌──────────────────────────────────────────────────────┐
│ Arachne: "必要なものを正確に見つけます" │
│ │
│ L1 プロジェクトツリー (AIが構造を把握するため) │
│ L2 login.ts (作業中のファイル) │
│ L3 http.ts, session.ts (検索 + │
│ 依存チェーン: login → session → http で発見) │
│ L4 config.ts (頻繁にアクセスされる、タイムアウト設定) │
│ │
│ → 30,000トークンの完璧にキュレーションされたコンテキスト │
└──────────────────────────────────────────────────────┘
 │
 ▼
 AIが正確な修正を生成 
```

**手動でのファイル選択不要。プロンプトエンジニアリング不要。ただ聞くだけ。**

---

### なぜArachneなのか？

- **98.5%のトークン節約** — 200万トークンの代わりに3万トークン。APIコストを大幅に削減
- **「中間の情報が失われる」問題を克服** — スマートな出力順序 (L1→L3→L4→L2) で重要なコードをAIが注目する位置に配置 ([研究に基づく](https://arxiv.org/abs/2307.03172))
- **外部依存ゼロ** — Docker不要、クラウド不要、APIキー不要。`npm install` するだけ
- **超高速** — 21ファイルを12msでインデックス。増分更新はサブ秒
- **超軽量** — 依存は3つだけ: `better-sqlite3`, `sqlite-vec`, `zod`。無駄なし
- **100%無料＆オープンソース** — Apache-2.0、隠れたコストなし、テレメトリなし
- **プラグ＆プレイ** — MCP設定を追加 → 完了。プロジェクトへのコード変更不要
- **多言語対応** — JS/TS、Python、Rust、Goのインポートチェーンを追跡
- **Ollamaはオプション** — Ollamaなしでも完璧に動作 (BM25検索)。Ollamaを追加すればセマンティック検索のボーナス付き

## 主な機能

| 機能 | 説明 |
|---------|-------------|
| **MCP標準** | Claude、Gemini、GPT、Ollama — あらゆるAIプロバイダーに対応 |
| **ローカルファースト** | すべてのインデックスはローカルSQLite。データは一切外部に出ない |
| **増分更新** | 変更されたファイルのみ再インデックス。サブ秒で更新 |
| **ハイブリッド検索** | BM25キーワード検索 + セマンティックベクトル検索 (Ollamaエンベディング) |
| **4層アセンブリ** | トークン予算内でのスマートなコンテキストページング |
| **依存グラフ** | JS/TS、Python、Rust、Goのインポートチェーンを追跡 |
| **バックアップ＆リストア** | SQLiteオンラインバックアップ、バックアップ内検索対応 |

## アーキテクチャ: 4層コンテキストアセンブリ

```
┌─────────────────────────────────────────────┐
│ トークン予算 (例: 3万) │
├────────────┬────────────────────────────────┤
│ L1: 固定 │ ファイルツリー概要 (10%) │
│ (常時) │ プロジェクト構造のスナップショット │
├────────────┼────────────────────────────────┤
│ L2: 短期 │ 現在のファイル + 最近 (20%) │
│ (コンテキスト)│ 今作業中のもの │
├────────────┼────────────────────────────────┤
│ L3: 関連 │ 検索 + 依存関係 (50%) │
│ (関連性) │ BM25 + セマンティック + 依存チェーン │
├────────────┼────────────────────────────────┤
│ L4: 予備 │ 頻繁にアクセスされるもの (20%) │
│ (バックアップ)│ よく使うファイル │
└────────────┴────────────────────────────────┘

出力順序: L1 → L3 → L4 → L2 (「中間の情報が失われる」問題を軽減)
```

## セマンティック検索 (オプション、ロックインなし)

Ollamaが利用可能な場合、Arachneはキーワードのみの検索から**ハイブリッド検索**にアップグレードされます:

```
BM25スコア (キーワード) ──┐
 ├── 重み付きマージ (α=0.5) → 最良の結果
コサイン類似度 ───────────┘
(nomic-embed-text 768次元)
```

- **sqlite-vec** によるSIMDアクセラレーション (AVX2/SSE2/Neon) KNNベクトル検索
- **768次元**エンベディング、Ollama `nomic-embed-text` 経由 — 100%ローカル実行
- **グレースフルデグラデーション**: Ollamaがない？BM25のみにフォールバック。**クラッシュゼロ。常に動作。**
- 設定で有効化: `embedding.enabled = true`
- ベクトルストレージ: チャンクあたり約3KB。5000チャンク = ディスク上わずか15MB

## 安定性: 104テスト、失敗ゼロ

Arachneはプロダクション向けに構築されています。あらゆるエッジケースがテスト済み:

| カテゴリ | テスト内容 |
|----------|---------------|
| SQLインジェクション | Bobby Tablesを含む5つの攻撃パターン |
| Null/空入力 | null、undefined、空文字列 → 安全な返却 |
| 巨大入力 | 10KBクエリ → クラッシュなし |
| 特殊文字 | Unicode、絵文字、正規表現文字 → 処理済み |
| Ollama切断 | 不正なエンドポイント → グレースフルBM25フォールバック |
| 冪等性 | 3回の再インデックス → 同じ結果 |
| 極端な予算 | 予算 0, 1, 100万 → すべて安全 |
| エッジtopK | topK = -1, 0, 99999 → クラッシュなし |
| スキーマ安全性 | 3回の初期化 → データ保持 |

```
Phase 1 (インデックス/検索):   15/15  PASS
Phase 2 (アセンブリ/依存):     26/26  PASS
Phase 3 (セマンティック/ハイブリッド): 19/19  PASS
安定性 (Reddit-proof):        44/44  PASS
─────────────────────────────────────
合計:                        104/104  ALL PASS
```

### MCP設定 (Claude Desktop / Cursor / etc.)

```json
{
 "mcpServers": {
 "n2-arachne": {
 "command": "node",
 "args": ["/path/to/n2-arachne/index.js"],
 "env": {
 "ARACHNE_PROJECT_DIR": "/path/to/your/project"
 }
 }
 }
}
```

## 設定

Arachneディレクトリに `config.local.js` を作成:

```javascript
module.exports = {
 projectDir: '/path/to/your/project',
 dataDir: './data',

 indexing: {
 autoIndex: true,
 maxFileSize: 512 * 1024, // ファイルあたり最大512KB
 },

 // セマンティック検索を有効化 (Ollamaが必要)
 embedding: {
 enabled: true, // デフォルト: false
 provider: 'ollama',
 model: 'nomic-embed-text',
 endpoint: 'http://localhost:11434',
 },

 assembly: {
 defaultBudget: 30000, // トークン
 },
};
```

## 使い方 (MCPツール)

Arachneは単一のMCPツール `n2_arachne` を以下のアクションで登録します:

| アクション | 説明 |
|--------|-------------|
| `search` | BM25キーワード検索 (+ セマンティック検索が有効な場合) |
| `assemble` | トークン予算内での4層コンテキストアセンブリ |
| `index` | プロジェクトファイルのインデックス/再インデックス |
| `status` | インデックス統計 + エンベディング状態の表示 |
| `files` | インデックス済みファイルの一覧 |
| `backup` | バックアップの作成/一覧/リストア |

### 例: コンテキストアセンブリ

```json
{
 "action": "assemble",
 "query": "HTTPリクエストのタイムアウトエラー処理",
 "activeFile": "lib/executor.js",
 "budget": 20000
}
```

## Soul / QLNとの連携

Arachneは単体でも十分に機能しますが、**Soul** (セッションメモリ) と **QLN** (ツールルーティング) と組み合わせるとさらに強力になります。

セットアップは簡単 — MCP設定に一緒に登録するだけ:

### Soul + Arachne の組み合わせ

```json
{
 "mcpServers": {
 "n2-soul": {
 "command": "node",
 "args": ["/path/to/n2-soul/index.js"]
 },
 "n2-arachne": {
 "command": "node",
 "args": ["/path/to/n2-arachne/index.js"],
 "env": {
 "ARACHNE_PROJECT_DIR": "/path/to/your/project"
 }
 }
 }
}
```

> **追加設定は一切不要！** 両方のサーバーを同じMCP設定に登録するだけで、AIが自動的に両方のツールを使用します。
> - `Soul` は過去のセッションの作業と決定を記憶
> - `Arachne` は正確なコードを見つけてAIに提供
> - 結果: AIは前回の続きからすぐに再開 — 「何をやっていたっけ？」がなくなる

### フルN2スタック (Soul + Arachne + QLN)

```json
{
 "mcpServers": {
 "n2-soul": {
 "command": "node",
 "args": ["/path/to/n2-soul/index.js"]
 },
 "n2-arachne": {
 "command": "node",
 "args": ["/path/to/n2-arachne/index.js"],
 "env": {
 "ARACHNE_PROJECT_DIR": "/path/to/your/project"
 }
 },
 "n2-qln": {
 "command": "node",
 "args": ["/path/to/n2-qln/index.js"]
 }
 }
}
```

> QLNを追加すれば、100以上のMCPツールがあっても、AIはQLNのセマンティックルーティングで必要なものだけを自動的に見つけて使用します。

## N2エコシステム — 組み合わせてさらに強力に

| パッケージ | 役割 | npm | 単体利用 |
|---------|------|-----|:----------:|
| **QLN** | ツールルーティング (1000以上のツール → 1ルーター) | `n2-qln` | Yes |
| **Soul** | エージェントメモリ＆セッション管理 | `n2-soul` | Yes |
| **Ark** | セキュリティポリシー＆コード検証 | `n2-ark` | Yes |
| **Arachne** | コードコンテキスト自動アセンブリ | `n2-arachne` | Yes |

> すべてのパッケージは**100%単体で動作**します。しかし組み合わせると、魔法が起きます:

### シナジー: どのように連携するか

```
ユーザー: "ログインのタイムアウトバグを直して"
 │
 ▼
┌─── QLN (ルーター) ───────────────────────────────────┐
│ 1000以上のツール → セマンティックルーティングが発見: │
│ → n2_arachne.assemble (コンテキスト) │
│ → n2_arachne.search (コード検索) │
│ トークンコスト: 1000以上ではなく2つのツール定義 │
└────────────────┬──────────────────────────────────────┘
 │
 ▼
┌─── Arachne (コンテキスト) ───────────────────────────┐
│ L1: プロジェクトツリー概要 │
│ L2: auth/login.ts (現在のファイル) │
│ L3: BM25 + セマンティック検索 → タイムアウト関連コード │
│ + 依存チェーン: login.ts → api.ts → http.ts │
│ L4: 頻繁にアクセスされる設定ファイル │
│ → 30Kトークンの完璧にキュレーションされたコンテキスト │
└────────────────┬──────────────────────────────────────┘
 │
 ▼
┌─── Soul (メモリ) ────────────────────────────────────┐
│ "前回のセッションで、Roseがapi.tsの47行目の │
│ 同様のタイムアウトを修正。決定: 30秒に増加。" │
│ → 過去のコンテキスト + 決定 + 引き継ぎメモ │
│ → KV-Cache: 即時セッション復元 │
└────────────────┬──────────────────────────────────────┘
 │
 ▼
┌─── Ark (セキュリティ) ───────────────────────────────┐
│ 生成コードにハードコードされた認証情報なし │
│ タイムアウト値はマジックナンバーではなく設定から取得 │
│ エラーハンドリングはプロジェクト規約に従う │
│ → コミット前のコード検証 │
└───────────────────────────────────────────────────────┘
```

### 単体 vs 組み合わせ

| シナリオ | 単体 | 組み合わせ |
|----------|------|----------|
| **トークン使用量** | AIが1000以上のツールをすべて見る | QLNがルーティング → AIは2-3ツールのみ |
| **コンテキスト品質** | AIがどのファイルが重要か推測 | Arachneが正確な関連コードを提供 |
| **メモリ** | AIは毎ターンすべてを忘れる | Soulが過去のセッション + 決定を記憶 |
| **コード安全性** | ガードレールなし | Arkがデプロイ前に検証 |
| **セットアップ** | 各ツールが独立して動作 | 追加設定不要 — 自動検出 |

### 実際の効果

- **QLN + Arachne**: QLNがリクエストをArachneにルーティング → Arachneが完璧なコンテキストを提供 → AIが初回で正確なコードを生成。「あのファイルどこだっけ？」がなくなる
- **Soul + Arachne**: Soulが前回のセッションの作業を記憶 → Arachneがそれらのファイルを優先的にインデックス → セッション間の継続性
- **Ark + Arachne**: Arachneがコードコンテキストを提供 → AIがコードを生成 → Arkがプロジェクトパターンに従っているか検証。出荷前にバグを検出
- **4つすべて**: AIが**すべてを記憶し**、**何でも見つけ**、**適切なツールを使い**、**ルールに従う**チームメンバーになる
